## Método

O CLI `claude mcp` desta versão **não possui** os subcomandos `list-resources` / `list-tools` / `call`
(só `add/get/list/remove/serve/add-json`). O smoke foi conduzido via **driver JSON-RPC direto pelo
pipe stdio** — exatamente o que um cliente MCP (Claude Desktop, Cursor) faz — rodando o servidor via
`tsx src/index.ts` (código-fonte atual da baseline).

**Env usado** (bypass local, igual ao `stdio-e2e.test.ts`):
`MCP_TRANSPORT=stdio · KEYCLOAK_BYPASS=true · SUPABASE_URL=https://stub.invalid · EMBEDDING_PROVIDER=stub`.

> ⚠️ **Não existe `.env` local** (só `.env.example`). A premissa do briefing ("usa GOOGLE_API_KEY local
> que você já tem") **não se aplica neste host** — não há credenciais Supabase nem chave de embedding.
> Por isso as chamadas que tocam Supabase/Google **não puderam ser exercitadas** com dados reais.

## 🔴 BUG ENCONTRADO (defeito em produção / baseline)

**`alexandria://metadata/workflows` falha em runtime — o resource nunca entrega.**

```
JSON-RPC error -32603 (Internal error)
ZodError em execution_matrix.workflow_{18,19,20}_backup:
  - total_duration_s : Required (esperado number, recebido undefined)
  - retry_policy     : Required (esperado string, recebido undefined)
  - sla_target       : Required (esperado number, recebido undefined)
```

**Causa raiz:**
- `WorkflowSchema` (`src/data/load-artifacts.ts:71-75`) marca `total_duration_s`, `retry_policy` e
  `sla_target` como **obrigatórios**.
- `execution_matrix: z.record(string, WorkflowSchema)` → **todo** workflow precisa cumprir.
- Os workflows **18/19/20_backup** em `AGENT_EXECUTION_MATRIX.json` têm só 5 campos
  (`workflow_id, workflow_name, trigger, agents, expected_daily_executions`) — **faltam os 3 acima**.
- → `loadExecutionMatrix` lança no parse → resource retorna -32603 a qualquer cliente.

**Por que passou no CI (28/28 verde)?**
O `stdio-e2e.test.ts` só faz `resources/read` de `metadata/agents`. **Nenhum teste lê
`metadata/workflows` nem `metadata/health` contra o JSON real.** O `health` passou no smoke por sorte
(schema permissivo `z.record(z.unknown())`); o `workflows` usa schema estrito e quebra.

**Impacto:** o resource `metadata/workflows` (um dos 2 entregáveis novos do M75 que eu commitei) está
**inutilizável** na baseline `975d404`. `metadata/health` e `metadata/agents` OK.

### Correção proposta (NÃO aplicada — aguarda OK do Arquiteto)

**Opção A (recomendada): tornar os 3 campos opcionais no `WorkflowSchema`.**
Workflows de backup legitimamente não têm SLA/duração/retry. Baixo risco, torna o resource tolerante
ao dado canônico:
```ts
total_duration_s: z.number().optional(),
retry_policy:     z.string().optional(),
sla_target:       z.number().optional(),
```
+ **regressão obrigatória:** adicionar ao `stdio-e2e.test.ts` um `resources/read` de
`metadata/workflows` **e** `metadata/health` (fecha o gap que deixou o bug passar).

**Opção B (rejeitada): preencher os 3 campos no JSON dos backups.**
Exigiria inventar valores (qual o `sla_target` de um job de backup?) — fabricar dado. Pior.

> Decisão sua: A ou B. É código de baseline (main), então é commit/PR novo — não toco sem seu OK.
